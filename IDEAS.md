# 献立ガチャ — 戦略・改善・ロードマップ

リリースブロッカー系（IAP実装、リストア、プラポリ等）は別セッションで対応中。
本ドキュメントではプロダクト全体の戦略を整理する。

---

## 現状評価

### 強み
- ガチャ×献立のコンセプトに独自性がある。競合が少ない
- Zustand + Stripe + Vercel のステートレス構成はシンプルで運用コストが低い
- レアリティ演出・Haptic・アニメーションのUX品質が高い
- Web版がVercelにデプロイ済みで、SEO・共有の基盤がある
- サーバーサイドAPIルート（/api/generate等）でAPIキーを安全に管理できている
- 旬の食材データ、代替食材データなどドメイン知識が実装済み

### 課題
- ユーザー獲得導線がほぼない。作って置いてあるだけの状態
- SEO施策がゼロ。Web版の最大の武器を活かせていない
- Free tierの制限がきつすぎて体験前に離脱するリスク
- Premium購入導線がSettings内のみ。発見しにくい
- プッシュ通知なし。翌日以降の再起動動機がない
- リワード広告が未実装（コンポーネントだけ存在）
- index.tsxが1394行。保守性に課題
- AI生成結果のキャッシュなし。毎回Claude API呼び出し

## 1. SEO対策（最優先・無料でできる）

現状Web版があるのにSEO施策がゼロ。これが最大の機会損失。

### 1-1. ランディングページ（`/lp`）のSEO最適化

**現状の問題：**
- `/lp`にmeta tags、h1、構造化テキストがほぼない
- OGPは`/api/ogp`で動的生成しているが、LP自体のSEO要素が不足

**ターゲットキーワードと月間検索ボリューム：**
- 「今日の献立」— 約10万/月
- 「献立 ランダム」— 数千/月
- 「献立 決められない」— 数千/月
- 「献立ガチャ」— ブランド指名（育てるべき）
- 「今日の晩ごはん」— 約5万/月
- 「簡単 献立」— 約3万/月

**対策：**
- `<title>`タグ：「献立ガチャ｜今日の献立をガチャで決めよう」
- `<meta name="description">`：「今日の献立に迷ったら、ガチャを回すだけ。メイン・副菜・汁物の3品をランダムで提案。レアリティ付きで楽しく献立が決まる無料アプリ。」
- `<h1>`：ページ内に適切な見出し階層を設置
- OGP tags（og:title, og:description, og:image, og:url）をLP内に直接埋め込む
- canonical URLの設定

### 1-2. レシピページのWeb公開（ロングテールSEO）

**現状の問題：**
- レシピはアプリ内でしか見れない。Web検索からのアクセス不可

**対策：**
- `/recipe/鶏の照り焼き` のようなURLでモックDB内レシピをWebに公開
- Google検索にインデックスさせる
- Schema.org Recipe構造化データを埋め込む：
  ```json
  {
    "@type": "Recipe",
    "name": "鶏の照り焼き",
    "cookTime": "PT30M",
    "recipeIngredient": [...],
    "recipeInstructions": [...],
    "nutrition": { "@type": "NutritionInformation", "calories": "350" }
  }
  ```
- Googleのリッチリザルト（レシピカード）に表示される → CTR大幅向上
- 各レシピページにアプリDLリンクとガチャへの導線を設置

**期待効果：**
- 「鶏の照り焼き 簡単」「豚の生姜焼き レシピ」等のロングテール流入
- 80レシピ × キーワード = 数百のインデックスページ
- レシピページ → アプリDL → ガチャ体験 の導線

### 1-3. 技術的SEO基盤

**対策：**
- `sitemap.xml` の作成・Vercelに静的配置
- `robots.txt` の作成（node_modules等を除外）
- Google Search Consoleへの登録・サイトマップ送信
- Vercelのプリレンダリング設定（SPAだとクローラーがJS実行できない場合がある）
- ページ速度の最適化（Lighthouse Core Web Vitals対応）

### 1-4. OGP画像の動的生成

**現状：** `/api/ogp`でテキストベースのHTML生成
**改善：** `@vercel/og`（Satori）でレシピ名+レアリティバッジ入りの画像を動的生成

**具体的な実装：**
- `/api/og-image?recipe=鶏の照り焼き&rarity=SSR` のようなエンドポイント
- レシピ名、レアリティバッジ（色分け）、カロリー、調理時間を画像に含める
- SSRは紫グロー、SRは金色等のビジュアル差別化
- X/Twitter, LINE, Facebookでの共有時にリッチなプレビュー表示

**期待効果：**
- SNS共有時のクリック率（CTR）が2〜3倍向上
- 「SSR出た！」投稿のバイラル効果増大

## 3. コンバージョン改善（Free → Premium）

### 3-1. Free tierのガチャ回数再設計

**現状の問題：**
- 初日10回→2-3日目5回→4日目以降3回/日
- 4日目以降3回は厳しすぎる。「体験する前に離脱」するリスクが高い
- 制限で課金を促すモデルは、ユーザーが十分にアプリの価値を感じた後でないと機能しない

**提案：**
- Free: **5回/日（固定）**
- 課金トリガーを「回数制限」から「品質差」に移行：
  - Free: モックレシピのみ（80件から選択）、履歴なし、広告あり
  - Premium: AI生成（無限バリエーション）、履歴+カロリー追跡、週間プラン、広告なし
- 回数制限で離脱させるより、使い続けてもらってAIの品質差を体験させる方がCVRが上がる

**検証方法：**
- A/Bテスト（3回/日 vs 5回/日）でDAU・CVR・チャーンを比較
- 初期はFirebase Remote Configなしでも、ハードコード切替で検証可能

### 3-2. 課金導線の強化

**現状の問題：**
- Premium購入導線がSettings画面の中にしかない
- ユーザーが課金機能の存在に気づかない

**対策：**

#### 結果画面でのアップセル
- 結果表示後に「AIならもっとあなた好みのレシピが出ます ✨」カード表示
- Free結果とAI生成結果の比較（「AIなら旬の食材を使った季節のレシピを提案」）
- ワンタップでCheckout画面に遷移

#### 機能タップ時のPaywall
- 週間モード・履歴・カロリーチャートをタップ時
- 現在のシンプルなAlertではなく、魅力的なPaywall画面を表示
- 「こんなことができます」のプレビュー + 「月額¥480で始める」CTA

#### 初月無料トライアル
- Stripeで `trial_period_days: 30` を設定可能
- 「まず1ヶ月無料で試す」のCTAは心理的ハードルが大幅に下がる
- トライアル後の継続率が重要指標になる

### 3-3. 価格戦略

**現状：** ¥480/月のみ

**提案：**
- **月額プラン：** ¥480/月（現行維持）
- **年額プラン：** ¥3,800/年（21%オフ、月あたり¥317）
  - 年額プランを「おすすめ」バッジ付きで表示
  - LTV向上（解約率低下 + 前払い）
- **1日体験：** ¥120/24時間
  - 特別な日（お正月、クリスマス等）のプロモ用
  - Non-recurring paymentsで実装（Stripeで設定済み）

## 6. 技術的改善

### 6-1. パフォーマンス・コスト最適化

| 課題 | 対策 | 効果 |
|------|------|------|
| AI生成が毎回Claude API呼び出し | よくある条件の組み合わせ結果をVercel KVにキャッシュ（TTL 24h） | API費用50%削減 |
| モックレシピ80件がクライアントバンドルに含まれる | `/api/recipes`エンドポイントに移動、必要分だけ取得 | バンドルサイズ削減 |
| OGP画像が静的テキスト | `@vercel/og`で動的画像生成 | SNS共有CTR向上 |
| 週間プランが7回API呼び出し | 1回のAPI呼び出しで7日分生成するプロンプト設計 | レスポンス時間短縮 |

### 6-2. AI生成キャッシュ戦略

**キャッシュキーの設計：**
```
{mealTime}:{genre}:{moods}:{cookingTime}:{servings}:{modes}
```

**ルール：**
- 同一条件の結果をVercel KVに保存（TTL: 24時間）
- 1条件あたり複数のバリエーション（5パターン）を保持
- リクエスト時にランダムに1つ返す → ユーザーには毎回新鮮に見える
- NG食材や履歴が絡む場合はキャッシュスキップ → AI直接呼び出し

### 6-3. セキュリティ確認事項

CLAUDE.mdのチェックリストに基づく確認ポイント：

| チェック項目 | 現状 | 対策 |
|-------------|------|------|
| APIキーのクライアント露出 | `/api/generate`経由で安全 | `src/lib/ai.ts`のフォールバック経路でクライアント直接呼び出しがないか再確認 |
| Stripeキーの管理 | 環境変数で管理OK | `STRIPE_SECRET_KEY`がクライアントバンドルに含まれないことを`dist/`で確認 |
| レート制限 | IP単位100回/24h | プロキシ経由の回避対策としてUser-Agent+フィンガープリント併用を検討 |
| JWT有効期限 | 要確認 | 長すぎる場合は短縮（推奨: 24時間）、リフレッシュトークン方式を検討 |
| `.env`の`.gitignore` | 要確認 | `.env`、`.env.local`がignoreされているか確認 |
| 課金バイパス | JWT検証あり | クライアントサイドのみの検証は不可。API側でもJWT検証を実施 |

### 6-4. 広告実装

**リワード広告（`RewardAdModal`が未実装）：**
- AdMob or ironSourceのリワード広告SDK導入
- 「広告を見てガチャ+1回」（1日最大3回）

## 8. 海外展開：Japanese Home Cooking Gacha

### 8-1. コンセプト
「日本の家庭料理を、海外のスーパーで買える食材で再現」するレシピガチャ。
国内向けの献立ガチャとは別モード or 別アプリとして展開。

### 8-2. なぜ刺さるか
- 日本食への関心は世界的に高いが「家で作る」ハードルが高い
- 専門食材（みりん、だし等）が手に入らない地域が多い
- 「海外の市販品で代用して本格的に作れる」レシピは需要があるのに供給が少ない
- ガチャ形式は言語を超えて直感的に楽しめるUI
- 「Japanese Cooking」はTikTok/YouTubeでトレンドジャンル

### 8-3. ターゲットユーザー
- 日本食に興味がある海外在住者（欧米・東南アジア）
- 海外在住の日本人（現地食材で和食を作りたい）
- 日本旅行後に「あの味を再現したい」層
- アニメ・日本文化ファン（「アニメ飯」への関心層）

### 8-4. 地域別の食材マッピング

レシピごとではなく、食材マスターに地域別代替を持たせる。

**代替例：**
| 本来の食材 | US代替 | EU代替 | どこでも |
|-----------|--------|--------|---------|
| みりん | honey + rice vinegar | sweet sherry | sugar + sake/white wine |
| だし | bonito flakes (Asian grocery) | kombu (health store) | mushroom broth |
| 薄口醤油 | regular soy sauce + salt | tamari + salt | — |
| 片栗粉 | cornstarch | cornstarch | cornstarch |
| 大葉 | basil + mint | basil + mint | — |
| みそ | miso paste (Whole Foods等) | miso paste | — |

### 8-5. レシピ構成
- 英語メイン（+ 日本語原名を併記：例 "Teriyaki Chicken (鶏の照り焼き)"）
- 各ステップに写真 or イラストが理想的
- 難易度表示：海外食材での再現難度（★1〜3）
- 「本場との味の近さ」指標（80%再現、95%再現 etc.）
- 「アジア食材店が近くにあれば…」の追加ティップス

### 8-6. 実装フェーズ
- **Phase 1:** 既存レシピDBに英語名と代替食材データを追加
- **Phase 2:** `expo-localization` + 辞書ファイル（日/英）で言語切替対応
- **Phase 3:** 地域設定UIと代替食材の自動切替
- **Phase 4:** AI生成時に地域・入手可能食材を考慮したプロンプト

### 8-7. 実装方針
**同一アプリの言語/地域切替を推奨。** 理由：
- コードベース1つで済む
- Stripeも多通貨対応済み（USD, EUR, GBP等）
- ストアのレビュー・レーティングが分散しない
- i18n基盤は他の多言語展開にも使い回せる

### 8-8. マネタイズとの相性
- 海外ユーザーは課金単価が高い（$4.99/月 ≒ ¥750）
- Amazon Affiliateで代替食材のリンクを貼れる（特にAmazon USのアジア食材）
- YouTubeやTikTokとのクロスプロモーション（レシピ動画→アプリ誘導）

### 8-9. 海外向けSEO
- 英語ランディングページ（`/lp/en`）
- ターゲットキーワード：「Japanese home cooking」「easy Japanese recipes」「Japanese meal plan」
- Schema.org Recipe構造化データを英語でも生成
- Reddit（r/JapaneseFood, r/Cooking）やQuoraでのコミュニティ参加

### 8-10. 検討事項
- 食材の入手性データはどう維持するか（ユーザー投稿？地域コミュニティ？）
- レシピの「再現度」は誰がジャッジするか（AI判定？ユーザー評価？）
- 日本のニッチ調味料（白だし、ポン酢等）の代替が難しいケースへの対応
